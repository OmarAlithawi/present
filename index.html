2.1 · Figure 1

Student Application Flow

The student’s journey from account creation through identity checks and mandatory payment to approval. After Submit the form locks — only an edit request from the directorate reopens it. The student is never rejected in their own flow; rejection sits with the directorate.

Identity
eKYC and هوية الأحوال verified at the Fill Application stage.
Payment
Mandatory before submit. Refunds, when applicable, are issued by the directorate.
Lock
After submit, the form is read-only. Only an edit request reopens it.
No Reject
Two outcomes only — approve, or edit request.
2.2 · Figure 2

School Director Flow

What the principal does when applications land in their queue. Two outcomes only — approve or edit request. The reject button has been removed; directors cannot turn an applicant away outright. Every action is logged with timestamp and reason. Approvals forward to the directorate; edit requests bounce back to the student.

Removed
The Reject button is gone — rejection is now a directorate-only action.

2.3 · Figure 3

Directorate Flow

The directorate operates two parallel queues, both visible to the directorate and the super admin. Routine school-approved requests pass eKYC and هوية الأحوال verification. When a school goes silent, days-elapsed triggers a notice to the school — the directorate does not take over the application. Two decision outcomes only — approve, or an edit request that bounces back to the school for fixes. No reject button anywhere in the pipeline.

Visibility
Directorate and super admin see every school’s applications — bottlenecks surface early.
Stall Trigger
Days elapsed triggers a notice to the school. The school remains the owner of the application.
Edit → School
An edit request from the directorate goes back to the school, not the student.
No Reject
Neither the school nor the directorate can reject — the only ways out are approve or edit.
Verification
eKYC and هوية الأحوال checked here, even on the routine pass.
Edge Case

What if the school doesn’t exist?

Two parallel ways a school enters the registry: the directorate adds it through their own portal, or the student requests it and the ministry plus the super admin co-approve.

  1. The directorate has its own portal to add the schools they oversee — no student request needed.
  2. If a student names a school that isn’t in the registry, they submit an “add school” request.
  3. The request reaches the ministry and the super admin in parallel for review.
  4. They check if the school already exists under a different name.
  5. If it does, the request is rejected with a reason.
  6. If not, they check whether it is named correctly.
  7. If correct, approve and add to the registry; otherwise, send an edit request to the student.
For Decision

Open questions

Tap a card to expand. These are the items raised on the working document that we need stakeholder input on.

Payment+

When does a refund trigger, and on what timeline?

Rejection is no longer in the flow — so refunds need a different trigger. Withdrawal before approval? Failed eKYC/هوية? Abandoned application after N days? And who covers gateway fees?

Identity+

What happens if eKYC or هوية الأحوال fails?

Hard block at Fill Application, or proceed with manual verification by the directorate? And does a failed check refund the payment automatically?

Account+

How hidden should “delete account” be?

Currently too prominent — should it require confirmation, sit under a settings sub-menu, or be gated behind directorate request? Any record retention rules to honour first?

Schools+

Should “add school” requests notify the local directorate?

Today the request goes to ministry + super admin. Should the directorate that owns the geographic area also see it — especially since they have their own portal to add schools?

Governance+

Who maintains the school registry over time?

Directorate adds via portal, ministry/super-admin approves student requests — but who is accountable for renames, mergers, and closures?

Compliance+

Is there any regulation we need to follow?

Need to identify the legal mandate — especially around payment handling, identity data retention, and minor protection.

Identity+

What happens if a student already exists from a prior year?

Carry over the prior record, or treat each year as a fresh application linked to the same student ID?

Eligibility+

Are there any categories other than martyrs’ sons?

Route A and Route B are defined for martyrs’ sons — do other protected groups need similar routing?

Lifecycle+

Can a student withdraw before approval?

If yes — before submit only, or also after director review? And how does that interact with the application-fee refund?

Lifecycle+

Data correction after approval?

Misspelled name on an already-approved record — who is allowed to fix it, and is the change audit-logged?

Account+

Multiple students sharing one account?

Should the parent account hold many student profiles? If yes, how is the application lock scoped — per student, or whole account?
